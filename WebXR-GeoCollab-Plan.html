What We're Building
The vision, scoped down to what is actually achievable
🎯
Core Vision

Three to four people wearing Meta Quest 3 headsets connect to a local Wi-Fi network. They open the browser on their headsets, navigate to https://192.168.x.x:3000, and enter a shared 3D space where they can see each other as avatars, talk in spatial audio, point at things, and look at your GIS terrain data together — all from one server PC with no internet required.

🗺️ Phase 1 — The Map Room MVP

  • Shared 3D space with GIS map draped on a flat terrain board
  • All users see each other as avatars with head/hand tracking
  • Spatial voice chat — talk over the map like a real briefing table
  • Pointer / laser tool to indicate positions on the map
  • Text chat panel for quick comms

🏔️ Phase 2 — 3D Terrain Walk Advanced

  • Actual Quantized Mesh terrain tiles served locally (from SRTM/PRSS-1 DEM)
  • CesiumJS-powered fly-through of real terrain
  • Multi-user synchronized camera — everyone sees the same view
  • Measure tool, annotation pins, area marking
  • UAV/drone overlay track playback on terrain
⚠️
Why Not a Native App?

Native Meta Quest apps require a developer account, sideloading pipeline, and PC tethering for upload. WebXR runs in the built-in Horizon OS browser with zero installation on the headset — you just type a URL. This is the right choice for a small team.

Key Insight: One PC Does Everything

The HTTPS web server, the NAF multiplayer signaling, and the terrain tile server are three small Node.js processes (or one combined) all running on the same PC. The Quest headsets are just browsers pointing to that PC's local IP.

3
Solving the HTTPS Problem
WebXR mandates HTTPS — here is the clean, permanent solution
⚠️
Why HTTPS is Required

Meta Quest's browser (Chromium-based) enforces that WebXR sessions and WebRTC (voice chat) can only be initiated from a secure origin — either localhost or a proper https:// URL. On a local IP (192.168.x.x), plain HTTP will not work. You will get "XR not supported" errors.

🔐 Solution: mkcert — Local Certificate Authority

mkcert is a zero-config tool that creates a local CA (Certificate Authority) and signs certificates for your local network IP. You install the CA root once on each device, and from that point on all your local HTTPS is fully trusted — no browser warnings.

# ── Step 1: Install mkcert on the SERVER PC (Windows or Linux) ── # On Linux: sudo apt install libnss3-tools curl -JLO "https://dl.filippo.io/mkcert/latest?for=linux/amd64" chmod +x mkcert-v*-linux-amd64 && sudo cp mkcert-v*-linux-amd64 /usr/local/bin/mkcert # ── Step 2: Create the local CA ── mkcert -install # This prints the CA root location. You'll need to export it. # ── Step 3: Create cert for your server's IP ── # Replace 192.168.1.100 with your PC's actual local IP mkcert 192.168.1.100 localhost 127.0.0.1 # Creates: ./192.168.1.100+2.pem and ./192.168.1.100+2-key.pem # ── Step 4: Export the CA root cert ── mkcert -CAROOT # prints location, e.g. /root/.local/share/mkcert/ # Copy rootCA.pem to a USB stick or shared folder
# ── Step 5: Install root CA on each Meta Quest 3 ── # Method A: via Android Debug Bridge (ADB) — recommended adb push rootCA.pem /sdcard/Download/rootCA.crt # Then on the Quest: Settings → Security → Install from storage → pick rootCA.crt # Method B: Host it on your server and navigate to it in the Quest browser # Quest browser will offer to install certificate when you download a .crt file
# ── Step 6: Use cert in Node.js HTTPS server ── const https = require('https'); const fs = require('fs'); const express = require('express'); const app = express(); const options = { key: fs.readFileSync('./certs/192.168.1.100+2-key.pem'), cert: fs.readFileSync('./certs/192.168.1.100+2.pem') }; app.use(express.static('public')); // Your WebXR HTML goes here https.createServer(options, app).listen(443, () => { console.log('🔒 HTTPS server running at https://192.168.1.100'); });
✅
One-time setup per headset

Once the root CA is installed on a Quest, it trusts your local server permanently. No re-setup needed unless you factory-reset the headset or change the server IP. The certificate mkcert generates is valid for 2 years.

4
Terrain Data Pipeline
From raw DEM (GeoTIFF) → Quantized Mesh tiles → CesiumJS in WebXR

📐 The Flow

Acquire DEM Data (GeoTIFF)

Use your SRTM data (30m), PRSS-1 sub-meter imagery, or any elevation raster for your AOI. Make sure it's in GeoTIFF format. Reproject to EPSG:4326 if needed using GDAL.

Run Cesium Terrain Builder via Docker

Use the Docker image tum-gis/ctb-quantized-mesh to tile your DEM into Quantized Mesh format. This runs entirely offline on your PC. For a 30m SRTM region, tiles up to zoom 12 are typically fine.

Generate layer.json Manifest

CTB also generates the layer.json file that CesiumJS needs to discover the tile structure. This file goes in the root of your terrain tile folder.

Serve Tiles via Node.js Static Server

Your Express server serves the tiles folder as static files. CesiumJS in each headset's browser requests tiles as needed, streaming only what's visible. No internet, no CDN.

CesiumJS Renders in WebXR

CesiumJS renders the terrain and you embed it inside the A-Frame/WebXR scene. Users can fly around and inspect the terrain in VR together.

# ── Terrain Processing Commands ── # 1. Build virtual dataset from multiple GeoTIFFs (if needed) gdalbuildvrt dataset.vrt /data/*.tif # 2. Reproject to EPSG:4326 (required by CesiumJS) gdalwarp -s_srs EPSG:32642 -t_srs EPSG:4326 dataset.vrt dataset-wgs84.vrt # 3. Add overview pyramid for multi-res rendering gdaladdo -r average dataset-wgs84.vrt 2 4 8 16 32 # 4. Run CTB inside Docker to generate Quantized Mesh tiles docker run -it --name ctb \ -v "./dem_data:/data" \ -v "./terrain_output:/terrain" \ tumgis/ctb-quantized-mesh # Inside the container: ctb-tile -f Mesh -C -N -o /terrain /data/dataset-wgs84.vrt ctb-tile -f Mesh -C -N -l -o /terrain /data/dataset-wgs84.vrt # Output: /terrain/0/ through /terrain/12/ + layer.json # 5. Serve terrain tiles via Express (in your main server.js) const cors = require('cors'); app.use('/terrain', cors(), express.static('./terrain_output')); # CesiumJS will request: /terrain/layer.json and /terrain/{z}/{x}/{y}.terrain # 6. In your WebXR HTML — CesiumJS terrain provider <script> const viewer = new Cesium.Viewer('cesiumContainer', { terrainProvider: new Cesium.CesiumTerrainProvider({ url: 'https://192.168.1.100/terrain' }), baseLayerPicker: false, geocoder: false }); </script>
💡
Phase 1 Shortcut

For the MVP Map Room phase, you don't even need terrain tiles. Just use a flat plane with a GeoTIFF rendered as a PNG texture draped on it. This is 10x simpler and lets you validate multi-user VR first before tackling the terrain pipeline.

Multi-User Networking
How 3–4 headsets synchronize position, voice, and state in real time

🔗 Networked A-Frame (NAF)

  • Syncs entity position and rotation 15 times/sec by default
  • Attach networked component to any A-Frame entity
  • Player avatars auto-created when users join
  • Interpolation built-in — smooth movement even at 15Hz
  • Works over WebSocket (reliable) or WebRTC P2P (low latency)

🎤 Voice Chat

  • Built into NAF via WebRTC data channel
  • Spatial audio — louder when avatars are close
  • Requires HTTPS origin (handled by mkcert)
  • Enable with: audio: true in scene config
  • No separate Jitsi or voice server needed
<!-- ── Core NAF Setup in your index.html ── --> <a-scene networked-scene=" room: geocollab-room-1; adapter: socketio; serverURL: https://192.168.1.100:3000; audio: true; debug: false; "> <!-- Avatar template (what other users look like) --> <a-assets> <template id="avatar-template"> <a-entity> <!-- Head --> <a-sphere radius="0.2" position="0 0 0" material="color: #4CC3D9"></a-sphere> <!-- Name tag --> <a-text value="Operator" position="0 0.35 0" align="center" scale="0.4 0.4 0.4"></a-text> <!-- Laser pointer --> <a-entity laser-controls="hand: right"></a-entity> </a-entity> </template> </a-assets> <!-- Local player — position/rotation synced to all others --> <a-entity id="player" camera position="0 1.6 5" networked="template:#avatar-template; attachTemplateToLocal:false;" networked-audio-source ></a-entity> <!-- The shared terrain/map environment --> <a-entity id="terrain-table" position="0 0 -3"></a-entity> </a-scene>
// ── naf-nodejs-server (signaling server) setup ── // Install: npm install naf-nodejs-server const { createServer } = require('https'); const { Server } = require('socket.io'); const NAFServer = require('naf-nodejs-server'); const fs = require('fs'); const httpsServer = createServer({ key: fs.readFileSync('./certs/key.pem'), cert: fs.readFileSync('./certs/cert.pem') }); const io = new Server(httpsServer, { cors: { origin: '*' } }); NAFServer(io); // Attach NAF room management to Socket.IO httpsServer.listen(3000, () => { console.log('🔗 NAF Signal Server: wss://192.168.1.100:3000'); });
🆕
Meta Quest Shared Spaces (Horizon OS v39+, June 2025)

If your team is in the same physical room, enable the experimental "Shared Spaces" flag in chrome://flags on each headset. This uses Meta's Shared Spatial Anchors so all headsets share the same real-world coordinate system — objects placed in VR align with the real room. Use PeerJS for the coordinate sharing signal as in Meta's official GitHub example.

Known Risks & Mitigations
Issues you will hit and how to avoid or fix them

🔴 HIGH CesiumJS + WebXR Integration

CesiumJS was not originally designed to render inside a WebXR canvas. Direct embed requires using CesiumJS as a Three.js renderer or rendering to a texture. This is the hardest technical challenge.

  • Use Three.js Cesium adapter (resium for React) or render Cesium to off-screen canvas → copy to WebXR texture
  • Phase 1–3 avoid this entirely (flat map on a plane)
  • Prototype with A-Frame 3D tiles loader (community component) first

🟡 MED Quest Browser Audio Permissions

WebRTC audio (voice chat) requires explicit microphone permission. Quest browsers prompt for this but users often accidentally deny it.

  • Add a clear on-screen "Mic Permission" button before entering VR
  • Quest settings: Apps → Browser → Permissions → Microphone → Allow
  • Test voice before every session

🟡 MED Large Tile Performance

Quantized Mesh tiles for a large AOI can overwhelm Quest's GPU if LOD settings are too aggressive. Quest 3 has 8 GB RAM shared between GPU and CPU.

  • Limit maximum zoom to level 12 (30m SRTM resolution)
  • Set CesiumJS maximumScreenSpaceError: 4 (less tiles loaded)
  • Pre-warm tile cache before VR session begins

🟡 MED NAF WebRTC on Local Network

WebRTC peer-to-peer sometimes fails when devices are behind NAT, even on LAN. STUN/TURN servers are usually only needed for internet setups, but can still be an issue.

  • Use WebSocket adapter instead of WebRTC for local LAN (slightly more latency, much more reliable)
  • All devices on same subnet — ensure router doesn't do client isolation
  • Disable "AP Client Isolation" on your Wi-Fi router

🟢 LOW WebGPU not yet in Quest Browser

As of Jan 2026, Quest's browser does not support WebGPU in WebXR mode (confirmed in Meta forums). This limits some advanced rendering.

  • Use WebGL 2.0 (fully supported) for rendering
  • CesiumJS has a WebGPU branch but WebGL 2 path is stable and adequate
  • Monitor Meta's forum for WebGPU+WebXR release

🟢 LOW Shared Spaces Colocation

The new Shared Spaces feature (June 2025) is experimental and per-room — if the last user exits WebXR, the shared space is lost and must be re-established.

  • Keep at least one headset in VR if shared anchors are needed
  • For briefing use cases, all enter together at session start
  • This is optional — NAF syncs avatars even without physical co-location
